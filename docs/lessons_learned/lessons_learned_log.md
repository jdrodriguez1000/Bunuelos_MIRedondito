# LOG DE LECCIONES APRENDIDAS - MI REDONDITO

## [2026-03-12] Refinamiento de Reportes MVP e Integridad
- **Technical**: Transformar resultados de validación en listas de micro-tests granulares. Esto permite una trazabilidad total "columna por columna" y "regla por regla", incrementando la confianza del usuario final en los resultados.
- **Process**: Persistencia local incondicional. El pipeline debe generar el `validation_report.json` local incluso si fallan las operaciones en la nube o si ocurre un error de integridad (`SECURITY_BREACH`), asegurando que el equipo siempre tenga un diagnóstico actionable.
- **Security**: Sincronización proactiva de hashes de contratos. Los cambios en el `data_contract.yaml` local invalidan la ejecución si no se sincronizan con Supabase. Se requiere un procedimiento o script de sincronización para evitar bloqueos del pipeline.
- **Process**: Gestión de limpieza de archivos. La eliminación recursiva de carpetas durante limpiezas generales puede borrar activos críticos (como `stage_builder`). Se debe validar la jerarquía de carpetas requeridas antes de ejecutar comandos de borrado masivo.

## [2026-03-13] Integración de CI/CD y Automatización de Lanzamientos (Release Please)
- **Security**: Autenticación de Acciones mediante PAT (Personal Access Token). Para que las acciones como `release-please` puedan crear Pull Requests y disparar otros workflows, se requiere un Token Clásico con scopes `repo` y `workflow`. El `GITHUB_TOKEN` estándar es insuficiente para estas tareas de escritura avanzada.
- **Technical**: Configuración de Release Please en modo "Manifest". Este enfoque es ideal para proyectos que evolucionarán a monorepos o esquemas multilingües (Python + Next.js). Permite llevar el conteo de versiones de forma independiente en un solo repositorio.
- **Process**: Codificación estándar UTF-8 para archivos de configuración. Se detectó que codificaciones como UTF-16 en archivos críticos como `requirements.txt` rompen el proceso de instalación en entornos Linux (GitHub Actions). La estandarización a UTF-8 (sin BOM) es un requisito innegociable de CI/CD.
- **Security/Process**: Equilibrio en Reglas de Protección de Rama. En proyectos con un único mantenedor, exigir aprobaciones manuales bloquea el flujo. La mejor práctica es exigir "Status Checks" (Quality Gate) exitosos pero permitir el merge al mantendedor principal para no detener la agilidad.
